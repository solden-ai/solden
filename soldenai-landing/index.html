Why now

Fragmentation grows with agents.

Back-office tool stacks keep growing. Most of the next additions will be agents.

An agent that does work is just another participant in a workflow that already lives across too many surfaces. Without a runtime, every agent is its own island, with its own state, and the coordination cost compounds.

Solden gives the agents somewhere to stand. The same record that holds human work holds agent work. The same audit chain that proves what a person did proves what an agent proposed. The same ownership trail that names who decided names which agent recommended and which human signed off.

The fragmentation isn’t a phase. It’s the new shape of work. The runtime is how you hold it.

The shape of the problem

Every back-office workflow lives in a patchwork of tools and one person’s head.

A finance team closing the month runs on a patchwork of tools. The vendor invoice in Gmail. The PO match in NetSuite. The approval routed in Slack. The accrual in a spreadsheet that lives on someone’s desktop. The audit binder in a shared drive.

The tools each do their job. The work between them lives nowhere. When the person who held it leaves, or forgets, or goes on vacation, the work goes with them.

That’s not an automation problem. That’s a coordination problem. And it’s getting worse, not better, as more of the work moves to agents that also need somewhere to stand.

How Solden works

One record per workflow.
Every move signed.

Every back-office workflow becomes a record. The record holds its own state, its own owner, what it’s waiting on, what broke, its history, and the policy that decided each call. It lives across your tools without belonging to any one of them.

Your tools render it. Solden holds it.

  1. The workflow becomes a record.

    Any back-office artifact enters wherever the team already works. Inbox, ERP queue, vendor portal, UBL feed. The runtime reads the unstructured input and lifts the typed fields onto a record. The record exists the moment work enters the building.

  2. Rules decide. The model explains.

    The cross-checks defined for the workflow type run against the typed record. Each gate is rule-based, versioned, and replayable. The model writes the explanation. It does not write the decision.

  3. Decisions render where the approver works.

    When a record needs a human call, the runtime surfaces it where the approver already lives. Gmail sidebar, Slack card, Teams card, NetSuite SuiteApp panel, SAP Fiori extension, and the next surface plugs in the same way. When approval lands, the runtime writes to the system of record with retry, idempotency, and a bounded override window.

  4. The audit chain seals itself.

    Every transition is signed with sha256. Each event hashes the event before it. Strip a row and the chain breaks. The proof writes itself as a side effect of doing the work. Six months later your audit committee reconstructs the entire workflow from one file.
